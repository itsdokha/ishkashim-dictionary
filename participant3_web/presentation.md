## Слайд 2 — Зачем этот проект? Кому полезен?

- Ишкашимский — **малоресурсный** памирский язык иранской группы (ISO 639-3: `isk`).
- У языка **нет общепринятой письменности**, лексика рассеяна по научным работам
  (Грирсон 1920, Пахалина 1959).
- Нет удобного цифрового словаря с быстрым поиском.

**Кому полезно:** студентам-лингвистам, исследователям памирских языков,
носителям, всем интересующимся.

> *Главная проблема малоресурсных языков — данные есть, но они разрозненны и
> неудобны. Мы собрали базовую лексику в одном месте с удобным поиском.*

## Слайд 3 — Что получилось (демо-скрин)

Статический сайт-словарь:
- строка поиска (ишкашимский ↔ русский);
- список результатов с подсветкой;
- карточка слова: перевод, варианты, примеры, этимология, комментарий, источники;
- фильтр по части речи; адаптивная вёрстка.

**159 словарных статей** (по плану минимум был 20).

> *Вот так это выглядит. Вводим слово на любом из языков — получаем список и
> подробную карточку. Сейчас покажем вживую.* → **демо**

## Слайд 4 — Как устроено: архитектура

```
data/dictionary.json      → данные (участник 1)
        ↓ fetch
participant2_js/search.js → поиск, нормализация, фильтры (участник 2)
        ↓ import
participant3_web/app.js   → интерфейс и сборка (участник 3)
        ↓
   сайт-словарь в браузере
```

Стек: **чистые HTML / CSS / JavaScript (ES-модули)** — без фреймворков и
сборщиков. Запуск — один `python3 -m http.server`.

> *Мы намеренно выбрали статический сайт: его легко запустить, легко защитить и
> в нём чётко видно разделение труда между нами тремя.*

## Слайд 5 — Задачи. Часть 1: данные и лингвистика

- Согласовали **формат словарной статьи** (контракт для всей команды).
- Собрали **159 статей** из научных источников.
- Поля: лемма (латиница с диакритикой), кириллица, часть речи, переводы
  (рус./англ.), варианты, примеры, **этимология**, комментарии, источники.
- Данные генерируются скриптом (`build_dictionary.py`) и проверяются
  валидатором — воспроизводимо, не «правится руками».

> *Концептуально-лингвистическая часть: унификация транскрипции, разметка
> частей речи, этимологии. Это самое трудоёмкое — данных по языку мало.*

---

## Слайд 6 — Задачи. Часть 2: поиск и логика

- Поиск **по ишкашимскому** (лемма, латиница, кириллица, варианты) и
  **по русскому** переводу.
- **Нормализация**: нижний регистр, снятие диакритики, `ё → е` — чтобы искалось
  «как набрали».
- **Фильтр по части речи**; поиск по подстроке.
- Корректная обработка **омонимов** (`wak` «один» и `wak` «там» — две статьи).

> *Техническая логика поиска вынесена в отдельный модуль и покрыта тестами —
> интерфейс просто её вызывает и ничего не знает о внутренностях.*

---

## Слайд 7 — Задачи. Часть 3: интерфейс и сборка

- Веб-страница: строка поиска, режимы, фильтр, список, карточка слова.
- **Подключён** модуль поиска участника 2 и данные участника 1 (без дублирования).
- Список частей речи и блок «О словаре» строятся **из данных**, не хардкодятся.
- **Адаптивная** вёрстка, подсветка совпадений, экранирование вывода.
- Оформлены **README** и эта презентация — продукт собран воедино.

> *Моя задача как участника 3 — превратить данные и поиск в готовый продукт,
> которым можно пользоваться, и собрать всё вместе.*

## Слайд 8 — Робастность: нестандартные случаи

| Ситуация | Что делает словарь |
|---|---|
| Пустой запрос | показывает весь словарь |
| Нет результатов | понятное сообщение + подсказка |
| Пустые примеры/этимология | блок не рисуется |
| Омонимы | показывает все статьи |
| Регистр / диакритика | нормализуются |
| Ошибка загрузки данных | инструкция про локальный сервер |

> *Мы специально проверяли «плохие» сценарии — это критерий робастности.*

## Слайд 9 — Сложности и проблемы

- **Малоресурсность языка**: источники старые, транскрипции расходятся —
  пришлось унифицировать вручную.
- **Нет письменности** → как искать? Решили: латиница без диакритики как главный
  ключ + кириллица + варианты.
- **`fetch` и `file://`**: данные не грузились при открытии файла напрямую —
  объяснили в README про локальный сервер и добавили понятную ошибку в UI.
- **Командный контракт формата**: чтобы три части стыковались, заранее
  зафиксировали формат статьи и не меняли его на ходу.
- **Омонимы и пустые поля** — частые «грабли», заложили их в логику и в UI.

> *Главный урок — без общего контракта формата три части не собрались бы. Мы
> договорились о нём в самом начале, и это спасло проект.*
